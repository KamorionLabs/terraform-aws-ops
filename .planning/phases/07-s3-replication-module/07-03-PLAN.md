<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| orchestrator -> source role | Orchestrator assumes the source role; the role's policy bounds what replication actions are possible |
| source role -> s3_replication role | `iam:PassRole` lets the source role hand the replication role to S3 services |
| S3 / batch service -> source data | The replication role is assumed by AWS services to read source / write destination objects |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-07-01 | Elevation of Privilege | `iam:PassRole` scope | mitigate | PassRole Resource is the specific `-s3-replication-role` ARN (no `*`) and constrained by `iam:PassedToService` listing exactly the two S3 service principals — source role cannot pass arbitrary roles (mirror EFS PassRoleToEFSReplication) |
| T-07-07 | Elevation of Privilege | combined trust policy over-grant | mitigate | Trust policy admits only `s3.amazonaws.com` and `batchoperations.s3.amazonaws.com` via `sts:AssumeRole`; no IAM-principal or account-root trust added |
| T-07-08 | Tampering | new IAM disrupts existing deployments | mitigate | `var.enable_s3` defaults false; all three resources count-gated so current consumers plan-noop (A3) |
| T-07-09 | Information Disclosure | over-broad source-data read by replication role | accept | Module-level Resources kept `*`/parameterized by design (generic opensource module); concrete bucket/KMS ARN tightening is Phase 8 client wiring (documented in RESEARCH Security Domain) |
| T-07-SC | Tampering | npm/pip/cargo installs | n/a | No package installs — Terraform HCL only |
</threat_model>
