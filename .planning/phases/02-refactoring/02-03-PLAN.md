**Step 1: Analyze refresh_orchestrator.asl.json** to identify the exact cluster switch sequence states.

- Note: ClusterSwitchSequence calls OTHER sub-SFNs (RenameCluster, etc.) so it needs their ARNs as input parameters (passed in the Input.StepFunctions field), NOT as templatefile variables.

**Step 2: Create cluster_switch_sequence.asl.json** in `modules/step-functions/db/`.
- Follow Phase 1 sub-SFN patterns: Comment with I/O contract, named Fail state (ClusterSwitchSequenceFailed), self-contained Catch on all Task states.
- Input contract (from research):
  ```json
  {
    "TargetClusterIdentifier": "string",
    "TmpClusterIdentifier": "string",
    "OldClusterIdentifier": "string",
    "OldInstanceIdentifierPrefix": "string",
    "TargetInstanceIdentifierPrefix": "string",
    "Options": { "RenameOldCluster": true, "DeleteOldCluster": false },
    "StepFunctions": {
      "Database": { "RenameCluster": "arn:...", "DeleteCluster": "arn:...", "EnsureClusterNotExists": "arn:...", "EnsureClusterAvailable": "arn:..." },
      "Utils": { "TagResources": "arn:..." }
    },
    "DestinationAccount": { "RoleArn": "arn:..." }
  }
  ```
- Output: {TargetClusterIdentifier, Status: "SwitchComplete", SwitchCompletedAt}
- Sequence (~10-12 states):
  1. RenameOldCluster (Task): Call RenameCluster sub-SFN for old cluster rename (if Options.RenameOldCluster is true via Choice)
  2. WaitAfterRenameOld (Wait): Brief stabilization wait
  3. CheckDeleteOldCluster (Choice): If Options.DeleteOldCluster
  4. DeleteOldCluster (Task): Call DeleteCluster or tag for later deletion
  5. TagOldCluster (Task): Call TagResources for old cluster
  6. RenameNewCluster (Task): Call RenameCluster sub-SFN for tmp -> target rename
  7. WaitAfterRenameNew (Wait): Brief stabilization wait
  8. EnsureNewClusterAvailable (Task): Call EnsureClusterAvailable
  9. RenameInstances (Pass or Task): Handle instance identifier renaming
  10. PrepareOutput (Pass): Format output
  11. SwitchComplete (Succeed)
  12. ClusterSwitchSequenceFailed (Fail): Named error
- Since ClusterSwitchSequence calls other sub-SFNs by ARN passed in Input, it uses `file()` (no templatefile variables needed).
- The sub-SFN ARNs are in `$.StepFunctions.Database.RenameCluster` etc. -- use JSONPath references, not template variables.

**Step 3: Register cluster_switch_sequence in DB module.**
- Add `cluster_switch_sequence = "cluster_switch_sequence.asl.json"` to `local.step_functions` in `modules/step-functions/db/main.tf`.
- No change to the `for_each` or resource -- it will be auto-deployed.
- The ARN will automatically appear in the merged output from `outputs.tf` (Plan 02-02 already set up the merge).

**Step 4: Update modules/step-functions/db/README.md.**
- Add ClusterSwitchSequence section with Input/Output contract, Behavior, Catch.
- Follow same format as EnsureSnapshotAvailable section added in Plan 02-02.
